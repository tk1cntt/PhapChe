# Phase 66 Wave 2: Partner Auth Enhancement

## Objective

Implement partner-specific authentication flow including dedicated login endpoint, invite system, and role-based permissions.

## Context

Phase 66 Wave 1 implemented basic partner APIs (dashboard, engagements) but partner auth relies on generic user login. Wave 2 adds dedicated partner auth with invite-based onboarding.

## Tasks

### Task 1: Partner Auth Service
**Files:** `src/lib/services/partner-auth-service.ts`

Implement `PartnerAuthService` with:
- `partnerLogin(email, password)` — Verify partner membership, return session with partner context
- `validatePartnerSession(session)` — Check user is active partner member
- `getPartnerPermissions(partnerId, userId)` — Get role-based permissions

```typescript
interface PartnerAuthResult {
  success: boolean;
  user?: User;
  partner?: Partner;
  partnerMember?: PartnerMember;
  permissions?: string[];
  error?: string;
}
```

**Verify:** Service exports all auth functions, TypeScript compiles

### Task 2: Partner Login API
**Files:** `src/app/api/partner/auth/login/route.ts`

Create `POST /api/partner/auth/login`:
- Accept `{ email, password }`
- Validate credentials via better-auth
- Check user is active PartnerMember
- Return session + partner context (partnerId, role, permissions)

```typescript
// Response
{
  success: true,
  user: { id, name, email },
  partner: { id, name, type },
  partnerMember: { role, permissions },
  session: { token, expiresAt }
}
```

**Verify:** `POST /api/partner/auth/login` returns 200 with partner context

### Task 3: Partner Invite Service
**Files:** `src/lib/services/partner-invite-service.ts`

Implement `PartnerInviteService`:
- `createInvite(partnerId, email, role, invitedBy)` — Generate invite token
- `acceptInvite(token, userId)` — Accept invite, link user to partner
- `revokeInvite(inviteId)` — Cancel pending invite
- `listPendingInvites(partnerId)` — Get all pending invites

```typescript
interface PartnerInvite {
  id: string;
  partnerId: string;
  email: string;
  role: 'admin' | 'specialist' | 'viewer';
  token: string;
  status: 'pending' | 'accepted' | 'revoked';
  invitedBy: string;
  expiresAt: Date;
}
```

**Verify:** Service exports all invite functions

### Task 4: Partner Invite API
**Files:** `src/app/api/partner/invite/route.ts`

Create partner invite endpoints:
- `POST /api/partner/invite` — Create new invite (partner admin only)
- `GET /api/partner/invite` — List pending invites
- `DELETE /api/partner/invite/[id]` — Revoke invite

Request body:
```json
{
  "email": "specialist@example.com",
  "role": "specialist"
}
```

**Verify:** Admin can create invite, invite appears in list

### Task 5: Partner Accept Invite API
**Files:** `src/app/api/partner/invite/accept/route.ts`

Create `POST /api/partner/invite/accept`:
- Accept `{ token }` (public endpoint, no auth required)
- Validate token not expired, not revoked
- Link user (from session) to partner as specified role
- Return success with partner context

**Verify:** New user can accept invite and access partner dashboard

### Task 6: Partner Members API
**Files:** `src/app/api/partner/members/route.ts`

Create `GET /api/partner/members`:
- List all member of partner organization
- Include user info, role, join date

```json
      "role": "specialist",
      "status": "active",
      "joinedAt": "2026-01-15"
```

**Verify:** Partner admin can view all members

### Task 7: Partner Member Management API
**Files:** `src/app/api/partner/members/[id]/route.ts`

Create member management endpoints:
- `PATCH /api/partner/members/[id]` — Update role, status
- `DELETE /api/partner/members/[id]` — Remove member

Request body:

```json
{
  "role": "admin",
  "status": "inactive"

```

**Verify:** Admin can change member role

### Task 8: Partner Permissions
**Files:** `src/lib/auth/partner-permissions.ts`

Define role-based permissions:

```typescript
const PARTNER_PERMISSIONS = {
  admin: ['manage_members', 'manage_engagements', 'view_requests', 'manage_requests', 'view_documents', 'manage_documents'],
  specialist: ['view_requests', 'manage_requests', 'view_documents', 'manage_documents'],
  viewer: ['view_requests', 'view_documents']
};
```

Create middleware `requirePartnerPermission(permission)`:
- Check session has partner context
- Check role has required permission

**Verify:** Middleware blocks unauthorized actions

## Acceptance Criteria

1. ✅ Partner can login via `/api/partner/auth/login` and receive partner context
2. ✅ Partner admin can invite new members via `/api/partner/invite`
3. ✅ Invited user can accept invite via `/api/partner/invite/accept`
4. ✅ Partner members list shows all organization members
5. ✅ Admin can change member roles
6. ✅ Role-based permissions enforced via middleware
7. ✅ All endpoints return proper error responses
