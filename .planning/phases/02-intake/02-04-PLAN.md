<threat_model>
## Trust Boundaries

| Boundary | Description |
|---|---|
| Browser form -> server actions | Customer-controlled requestId, answer fields, and file input cross into server mutations. |
| Env/session source -> Prisma user lookup | `APP_SESSION_USER_ID` selects current dev/server actor; DB active membership defines tenant and roles. |
| Route search param -> server render | `requestId` in `/intake?requestId=` is untrusted and must pass `canAccessRequest`. |
| Prisma stored intake/upload metadata -> review UI | Sensitive saved answers and filenames render only after server-side RBAC. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|---|---|---|---|---|
| T-02-gap-01 | Spoofing | `src/lib/security/session.ts` | mitigate | Remove hard-coded demo actor; require `APP_SESSION_USER_ID`; load active user/membership from Prisma before returning AppSession. |
| T-02-gap-02 | Tampering | `src/app/intake/page.tsx` hidden requestId forms | mitigate | Treat hidden requestId as untrusted; page load and existing actions use `requireAppSession` + `canAccessRequest`/service guards before reads/writes. |
| T-02-gap-03 | Information Disclosure | `/intake?requestId=` review data | mitigate | Only load intakeSubmission/vaultFiles after `canAccessRequest(session, requestId)`; never expose storageKey/public URL. |
| T-02-gap-04 | Repudiation | Intake mutations/audit actor | mitigate | Session actor comes from DB user id; existing services record audit with `session.userId`; no demo-customer audit pollution. |
| T-02-gap-05 | Information Disclosure | Upload file metadata review | mitigate | Render filename and size only; no storageKey, hash, URL, or file content. |
</threat_model>
