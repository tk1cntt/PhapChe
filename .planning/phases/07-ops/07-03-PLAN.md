<threat_model>
## Trust Boundaries
| Boundary | Description |
|---|---|
| URL path `requestId` -> server page | User-controlled request identifier selects timeline scope. |
| Server page -> ops service | Page asks for request timeline with privileged admin context. |
| Timeline DTO -> rendered HTML | Audit/workflow records are exposed to an admin screen. |

## STRIDE Threat Register
| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|---|---|---|---|---|---|
| T-07-03-01 | Tampering | dynamic `requestId` param | High | mitigate/block | Treat `requestId` as untrusted; pass to `getOpsRequestTimeline` which validates request/workspace/admin access before returning data. |
| T-07-03-02 | Information Disclosure | timeline rendering | High | mitigate/block | Render only safe DTO fields returned by service; never add Prisma queries or select raw review/document/vault/intake fields in the page. |
| T-07-03-03 | Information Disclosure | unauthorized request timeline | High | mitigate/block | Call `requireAppSession` and `getOpsRequestTimeline`; render existing unauthorized/error behavior without dumping data or stack traces. |
| T-07-03-04 | Integrity | event chronology | Medium | mitigate | Render service-sorted reverse chronological timeline and include kind badges (`audit`/`workflow`) so audit and workflow events are not conflated. |
</threat_model>
