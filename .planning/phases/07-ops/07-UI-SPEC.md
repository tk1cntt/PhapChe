## Spacing Scale

Declared values (multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline gaps inside compact badges or metadata pairs |
| sm | 8px | Navigation item spacing, badge groups, table row inline actions |
| md | 16px | Default card padding on mobile, form field gaps, table cell horizontal rhythm |
| lg | 24px | Card padding on desktop, section-level grouping |
| xl | 32px | Main content vertical rhythm between dashboard sections |
| 2xl | 48px | Major page breaks only when separating cockpit summary from drill-in content |
| 3xl | 64px | Reserved for page-level spacing; avoid inside dense admin tables |

Exceptions:
- Interactive controls must keep a minimum hit target of 40px using existing `Button` `min-h-10`.
- Dense table cells should use 8px or 16px vertical padding in new Phase 7 code.
- Timeline connector dots may be 8px, and row spacing must still follow 8/16/24px rhythm.

## Typography

Use exactly these four sizes and two weights for Phase 7.

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Caption / table header | 12px | 600 (semibold) | 1.4 | Uppercase table headers, small metadata labels |
| Small body / badges | 14px | 400 or 600 | 1.4 | Table metadata, filters, badge text, nav labels |
| Body | 16px | 400 (normal) | 1.5 | Section descriptions, form inputs, empty/error states |
| Heading | 20px | 600 (semibold) | 1.2 | Card titles, workload/table section headings, page-level heading when custom text is needed |

Weight contract:
- Normal: `400` for body copy, timestamps, safe metadata summaries.
- Semibold: `600` for headings, labels, badges, CTA buttons.
- Do not introduce bold `700` in Phase 7.

## Page Structure

### Admin Ops Dashboard (`/admin/ops`)

Use existing `AdminShell` and `PageHeader`. Plain cards/tables are required; do not add charting unless a future phase explicitly approves it.

Primary focal point: the `Tổng quan vận hành` metric grid is the first visual anchor. Within it, `Cần chú ý theo aging` gets warning emphasis before the eye moves to the filtered request list.

```
AdminShell
  PageHeader
    title: "Vận hành"
    description: ops visibility + safe audit note

  Section: Tổng quan vận hành
    Grid of Card components
      - Tổng hồ sơ
      - Theo trạng thái
      - Đang chờ review
      - Cần chú ý theo aging
    Each card: small label, 20px metric value, 14px secondary explanation

  Card: Bộ lọc hồ sơ
    Form submitted via query string/searchParams
      - Customer/workspace select
      - Matter type select
      - Status select
      - Specialist select
      - Reviewer select
      - Date from / date to inputs
      - Button: "Áp dụng bộ lọc"
      - Secondary link/button: "Xóa bộ lọc"
    Filters compose with AND semantics server-side.

  Section: Danh sách hồ sơ
    Table columns:
      - Mã hồ sơ
      - Khách hàng / Workspace
      - Matter type
      - Trạng thái
      - Specialist
      - Reviewer
      - Tuổi hồ sơ
      - Tuổi trạng thái hiện tại
      - Thao tác: "Xem timeline"

  Section: Workload chuyên viên và reviewer
    Two Card/Table blocks or one grouped table:
      - Person
      - Role (specialist/reviewer)
      - Active requests
      - Pending review
      - Revision required
      - Oldest active age

  Optional when request selected: Timeline audit
    Card/Table or vertical timeline for one request
      - Time
      - Actor
      - Kind (audit/workflow)
      - Action/status change
      - Target identifier
      - Reason
      - Correlation ID
      - metadataSummary
```

### Request Timeline Drill-in

Preferred MVP interaction: open a request-specific timeline from `Xem timeline`. Planner may choose `/admin/ops/[requestId]` or `/admin/ops?requestId=...`; UI contract is the same.

```
PageHeader: "Timeline audit" + request code
Card: Safety note
Table/Timeline rows in reverse chronological order
Back link: "Quay lại vận hành"
```

Interaction contract:
- Filters must be reflected in URL query params so refresh/share keeps the same view.
- Filter submission is a normal server-rendered navigation, not client-only filtering.
- Empty states must render inside the relevant card/table area, not as a blank page.
- Timeline must never reveal raw legal content, reviewer-only comments, document text, vault storage keys, or file contents.
- Metrics and filtered request list must use the same normalized server-side filter semantics.

|-----------|---------|--------|
| `AdminShell` | Admin page frame and navigation | default; add ops nav item labeled `Vận hành` if route is introduced |

| Timeline row | Audit/workflow chronology | audit event, workflow transition, missing optional metadata |
| Empty state block | No data for dashboard/list/timeline | default informational state |
| Alert/note text | Safety and error messages | info, error |

Do not introduce modal dialogs, destructive confirmation, chart canvas, CSV/PDF export, saved-view chips, fuzzy search input, or employee scoring visuals in Phase 7.

## Data Display Rules

| Data | Display Contract |
|------|------------------|
| Counts by status | Display as cards/table counts from backend data; no hard-coded fixtures |
| Assignee/reviewer counts | Display person label plus role; use `Chưa phân công` when absent |
| Aging | Human-readable compact form: `2 ngày`, `5 giờ`, `Vừa cập nhật`; avoid SLA breach language unless backend provides policy |
| Date/time | Use consistent admin format: `YYYY-MM-DD HH:mm` or existing project formatter if present |
| Audit metadata | Show `metadataSummary` only; truncate visually after one line if long, but keep full safe text available only if existing pattern supports it |
| Actor | Show email/name if safe; fallback `system` |
| Target | Show stable identifiers such as `request:<id>` or request code; never storage keys |
| Reason | Show reason text when it is already part of workflow/audit safe fields |
