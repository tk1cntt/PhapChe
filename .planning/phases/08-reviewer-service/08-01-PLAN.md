    async function loadReviewForActor(reviewId: string, session: AppSession) {
      const review = await prisma.review.findUnique({
        where: { id: reviewId },
        select: {
          id: true, workspaceId: true, requestId: true, documentId: true, reviewerId: true,
          documentVersionId: true, status: true, generalComment: true,
          request: { select: { assignedReviewerId: true, status: true, workspaceId: true } },
        },
      });
      if (!review) throw new Error('REVIEW_NOT_FOUND');
      const isAssignedReviewer = review.reviewerId === session.userId && review.request.assignedReviewerId === session.userId;
      const isAdmin = session.roles.includes('coordinator_admin') || session.roles.includes('super_admin');
      if (!isAssignedReviewer && !isAdmin) throw new Error('FORBIDDEN');
      return review;
    }

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → Server action | Untrusted FormData crosses here; `actions.ts` (Plan 02) parses and forwards to service. Service MUST re-validate every permission and state. |
| Reviewer → Service | Reviewer identity is asserted via `session.userId`; service must verify `assignedReviewerId === session.userId` or admin role. |
| Service → DB | $transaction boundary — must keep DocumentVersion update, Review update, and audit-event insert atomic so we cannot end up with `DocumentVersion.status='final'` but no audit trail. |
| Service → Workflow | `transitionRequestStatus` is called outside the $transaction (uses `prisma`, not `tx`) — accepted race window matches `submitForReview` pattern; mitigation is that `transitionRequestStatus` itself writes a `WorkflowTransition` row and audit event in its own transaction. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-01 | Spoofing | review-service startReview / approve / reject | mitigate | `loadReviewForActor` re-verifies `reviewerId === session.userId` AND `request.assignedReviewerId === session.userId` on every call; admin roles also allowed. |
| T-08-02 | Tampering | answerChecklistItem | mitigate | Each answer is `upsert`ed by `(reviewId, checklistItemId)`; UI cannot forge answers for a closed review because `loadReviewForActor` requires `status === 'in_progress'`. |
| T-08-03 | Repudiation | approveReview / rejectReview | mitigate | Every state change writes a `recordAuditEvent` with `action: 'review.approved' | 'review.rejected' | 'review.started' | 'review.checklist_answered'`, `targetType: 'REVIEW'`, `targetId: reviewId`, plus safe metadata. |
| T-08-04 | Information Disclosure | review-service | mitigate | `metadataSummary` is capped at 500 chars by `recordAuditEvent` (audit.ts:65) and contains only IDs + counts — never `generatedContent` or `generalComment`. D-06 enforced. |
| T-08-05 | Denial of Service | review-service | accept | Service runs only on authenticated reviewer actions; no anonymous endpoint. No rate limiting at this layer. |
| T-08-06 | Elevation of Privilege | approveReview / rejectReview | mitigate | Service throws `FORBIDDEN` if `assignedReviewerId !== session.userId`; reviewer cannot approve a request assigned to a different reviewer. Workflow `canTransitionRequestStatus` adds second-layer check (request-workflow.ts:59-61). |
| T-08-07 | Incomplete Validation | approveReview | mitigate | Hard-coded required-item list is checked before any state change; throws `CHECKLIST_NOT_COMPLETE` (no half-approved state). Test 3 covers. |
| T-08-08 | Incomplete Validation | rejectReview | mitigate | Empty `generalComment` throws `REJECT_COMMENT_REQUIRED` before any state change. Test 4 covers. |
</threat_model>
