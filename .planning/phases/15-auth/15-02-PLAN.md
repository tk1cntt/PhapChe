<task type="auto">
  <name>Task 3: Adapt requireAppSession() to use Better Auth</name>
  <files>src/lib/security/session.ts</files>
  <read_first>@src/lib/security/session.ts, @auth.ts</read_first>
  <action>
    Rewrite `/mnt/d/PhapChe/src/lib/security/session.ts` to use Better Auth while preserving the exact same exports and return type.

    The new implementation:
    ```typescript
    import { auth } from "@/auth";
    import { headers } from "next/headers";
    import { prisma } from "@/lib/prisma";

    export type AppRole = 'customer' | 'specialist' | 'reviewer' | 'coordinator_admin' | 'super_admin';

    export type AppSession = {
      userId: string;
      activeWorkspaceId?: string | null;
      roles: AppRole[];
    };

    export async function requireAppSession(): Promise<AppSession> {
      const session = await auth.api.getSession({ headers: await headers() });
      if (!session?.user?.id) throw new Error('UNAUTHENTICATED');

      const userId = session.user.id;
      const user = await prisma.user.findFirst({
        where: { id: userId, isActive: true },
        select: {
          id: true,
          memberships: {
            where: { isActive: true, workspace: { isActive: true } },
            select: { workspaceId: true, role: true },
            take: 1,
          },
        },
      });

      const membership = user?.memberships[0];
      if (!user || !membership) throw new Error('UNAUTHENTICATED');

      return {
        userId: user.id,
        activeWorkspaceId: membership.workspaceId,
        roles: [membership.role],
      };
    }
    ```

    CRITICAL design decisions:
    - `auth.api.getSession()` validates the session server-side (not just cookie check) — this is the REAL auth check
    - `headers()` must be called inside the function, not at module scope — Next.js 16 requires dynamic header reading in request context
    - The user + membership query is IDENTICAL to the current stub — ensures same behavior
    - The return type is EXACTLY `{ userId, activeWorkspaceId, roles }` — 23 call sites receive unchanged data
    - Error message 'UNAUTHENTICATED' is same string as before — any error handling that checks this string continues working

    Do NOT modify the import path or export names. The 23 callers import `{ requireAppSession }` from `@/lib/security/session` and that MUST NOT change.
  </action>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-15-02 | Elevation of Privilege | proxy.ts | mitigate | Proxy checks session existence; actual auth validation happens in `auth.api.getSession()` inside requireAppSession() — proxy is optimistic only, not the real auth gate |
| T-15-03 | Spoofing | auth.ts (email/password) | mitigate | Better Auth uses scrypt hashing, CSRF protection by default |
| T-15-04 | Tampering | session.ts | mitigate | `auth.api.getSession()` validates token server-side, not just cookie parse; session rotation handled by Better Auth |
| T-15-05 | Information Disclosure | proxy.ts matcher | accept | Matcher excludes public paths (/sign-in, /api/auth, static assets) — unauthenticated access to these does not leak data |
</threat_model>
