# Plan 20-01 — i18n Foundation and Customer Locale Coverage

<objective>
Complete Phase 20 internationalization infrastructure: use `next-intl` with locale-prefixed routes for `vi`, `en`, `zh`, `ja`; keep `/sign-in` and `/api/auth` unprefixed; add locale provider under `[locale]`; make the language switcher preserve page path; localize priority customer-facing route entry points; and add Playwright e2e coverage for locale routing/language switching.
</objective>

<threat_model>
## Security Threat Model

| Threat | Severity | Mitigation in this plan |
|--------|----------|--------------------------|
| Locale prefix bypasses protected route checks | high | Keep middleware auth guard for localized app routes and preserve existing server-side `requireAppSession()` checks in pages/services. |
| Auth callback/API route accidentally localized | high | Middleware matcher and logic must exclude `/sign-in`, `/api/auth`, and static assets. |
| Language switcher strips path/query and sends user to wrong sensitive workspace/request context | medium | Switcher must preserve logical pathname after replacing only locale prefix. |
| Missing translation key crashes user-facing page | medium | Ensure all keys used by new localized customer/intake UI exist in all four message files or avoid new key usage. |
</threat_model>
