  <action>
Create four standards documents:

**src/docs/API_STANDARDS.md:**
- Response envelope pattern: `{data, meta?}` for success, `{error, detail?}` for errors
- HTTP status codes mapping (200, 201, 400, 401, 403, 404, 500)
- Pagination pattern: cursor-based with `meta.cursor`, page-based with `meta.page`, `meta.pageSize`, `meta.total`
- Error codes enumeration: FORBIDDEN, NOT_FOUND, VALIDATION_ERROR, etc.
- Naming conventions: kebab-case for endpoints, camelCase for request body

**src/docs/CODE_STANDARDS.md:**
- File naming: PascalCase for components, camelCase for utilities, kebab-case for pages
- Directory structure: feature-based with components/, api/, lib/, types/ per feature
- Import order: React > external libraries > internal packages > relative imports
- TypeScript rules: strict mode, no `as any` in new code, explicit return types for exported functions

**src/docs/SERVICE_LAYER.md:**
- Service responsibilities: business logic, database access, external API calls
- Service location: `src/lib/{domain}/` with index.ts barrel exports
- Service patterns: input validation, error handling, audit logging, transaction management
- Anti-patterns: no business logic in components, no direct Prisma calls in API routes

**src/docs/I18N_RULES.md:**
- Translation scope: all user-facing text in components (labels, buttons, messages, errors)
- Non-translated: internal logs, error codes, technical identifiers, debug output
- i18n key format: `namespace.key` (e.g., `UserDashboard.statTitle`)
- Fallback: provide English default, handle missing keys gracefully
- Implementation: use `useTranslations()` hook, `t()` function for dynamic strings, static text for rare/singular use
  </action>
