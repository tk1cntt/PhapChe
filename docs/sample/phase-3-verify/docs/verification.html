1 Philosophy: verification ≠ testing

This architecture is built on a single mental model:

Verification is runtime observation at the surface.

You build the app, run it, drive it to where the changed code executes, and capture what you see. That capture is your evidence. Tests and typechecks are CI's job. A verifier's job is to confirm the real artifact behaves — at the boundary where a user (human or programmatic) meets it.

For a CLI, the surface is the terminal. For a server, it's the socket. For a React app, the surface is the rendered DOM. So this system makes the DOM itself machine-readable, gives every component a way to be mounted in isolation in a known state, and exposes a structured API for reading verdicts.

ConcernUnit testingVerification (this system)
What runsA function, in Node, with mocksThe component, in a real DOM, with real state
What's checkedReturn values / mock callsWhat's actually rendered & how it responds
Who consumes itCICI and dashboard and AI agent — same code path
Failure mode caughtLogic bugLogic bug, integration bug, contract drift, a11y regression
Verdict shapepass / failPASSFAILBLOCKEDSKIP

2 Anatomy of the system

Reading the diagram: components emit DOM contracts; specs and verifiers register into a shared registry; the runner is the single execution path that mounts a fixture, runs verifiers, and produces a structured VerifyResult. Four consumers all call the runner — they differ only in how they display or assert on the result.

3 The DOM contract (data-verify-*)

The lowest layer. Every component spreads verifyAttrs({...}) onto its root element, emitting a set of data-verify-* attributes that describe its semantic state — not its styling, not its internals. 

// features/todos/TodoApp.tsx
<section
  className="todo-app"
  {...verifyAttrs({
    unit: "TodoApp",
    total: t.stats.total,   done: t.stats.done,   active: t.stats.active,
    filter: t.filter,      visible: t.visible.length,
  })}
>

Which renders as:

<section class="todo-app"
  data-verify-unit="TodoApp" data-verify-total="3" data-verify-done="1"
  data-verify-active="2" data-verify-filter="all" data-verify-visible="3">

Why this matters

The DOM is the surface. Agents and verifiers read readContract(root) → {unit, total, done, active, filter, visible} instead of reaching into React fibers or guessing from CSS classes. You can rewrite the entire component internals — switch state libs, restyle, refactor — and as long as the contract holds, every verifier still passes.

The helper lives in src/verify/core/contract.ts and provides verifyAttrs() (write), readContract() (read the root contract), and readAllContracts() (read every contract-bearing descendant — useful for lists/features).

4 Verifiable units, fixtures & invariants

A VerifiableUnit is the unit of modularity: a single component (kind: "component") or a whole feature slice with its own state (kind: "feature"). Each spec file in src/verify/specs/ registers exactly one. 

// src/verify/specs/TodoItem.verify.ts (abridged)
registerUnit<TodoItemProps>({
  id: "TodoItem",
  title: "TodoItem",
  kind: "component",
  render: (props) => createElement(TodoItem, props),
  propsSchema: z.object({ todo: TodoSchema, onToggle: z.function(), onRemove: z.function() }),

  fixtures: [
    { id: "active",     props: { todo: { text: "Buy milk", done: false, ... } } },
    { id: "done",       props: { todo: { text: "Ship it",  done: true,  ... } } },
    { id: "empty-text", probe: true, props: { todo: { text: "", ... } } },
    { id: "long-text",  probe: true, props: { todo: { text: "x".repeat(500), ... } } },
  ],

  invariants: [
    { id: "done-attr-matches-prop",
      check: ({ contract, props }) => contract.done === String(props.todo.done) },
    { id: "checkbox-state-matches",
      check: ({ root, props }) => root.querySelector("input[type=checkbox]").checked === props.todo.done },
    { id: "visible-text-never-empty",
      check: ({ root }) => root.querySelector(".todo-text").textContent.trim().length > 0 },
  ],
})

Fixtures

A Fixture is a named, reproducible rendering configuration. It has props and an optional imperative act(ctx) step that runs after mount and before verification — letting a fixture express "render, then click X, then verify." 

// from todos.feature.verify.ts — a behavioral fixture
{
  id: "add-then-toggle",
  props: {},
  act: async (ctx) => {
    ctx.type("input[type=text]", "Ship it");
    ctx.click("button[type=submit]");
    await ctx.wait(16);
    ctx.click("li.todo-item input[type=checkbox]");
  },
}

🔍 Probes are required

Fixtures marked probe: true are adversarial / off-happy-path cases — empty text, 500-char text, whitespace-only submit, inconsistent counts. The rule of thumb: "a check list that's all ✅ and no 🔍 is a happy-path replay." The CI matrix test (matrix.test.ts) fails any unit that declares zero probe fixtures — you cannot ship a unit that only tests the happy path.

Invariants

An Invariant is a named predicate over the mounted DOM: (ctx) => boolean | string. true holds; false or a string violates (string becomes the diagnosis). Invariants receive:

rootthe live HTMLElement the unit is mounted in
propsthe fixture's props
contractthe parsed data-verify-* map — usually all you need
fixturethe fixture object (for conditional logic)

Use onlyFixtures: ["id", ...] to scope an invariant to specific fixtures (e.g. "after adding, total is 1" only makes sense for the add-then-verify fixture).

Component vs feature units

Component unit

TodoItem, TodoInput, TodoList, TodoStats. Stateless from the spec's perspective — fixtures pass props, invariants check the contract reflects them. Tests one piece in true isolation.

Feature unit

todos.feature = the whole TodoApp. Has real state. Fixtures lean on act() to drive interactions. Invariants check coordination: "TodoStats's contract agrees with TodoApp's contract", "visible count matches filter", "adding a todo updates list AND stats".

5 Pluggable verifiers

A Verifier is a kind of check. It's independent of any component — it inspects whatever unit×fixture is mounted and returns structured Check[]. Four ship in this demo:

VerifierWhat it checksFail means
schemaFixture props parse against the unit's Zod propsSchema (incl. .refine() rules)The fixture is lying about valid input, or a prop type drifted
invariantsRuns every declared invariant predicate against the live DOMA contract claim doesn't hold
dom-contractA data-verify-* contract is present and self-identifies (data-verify-unit)The surface isn't machine-readable — agents would have to guess
a11yButtons named, inputs labeled, images have altAccessibility regression

The interface is tiny: 

interface Verifier {
  id: string
  description: string
  run: (ctx: { unit, fixture, root, contract }) => Check[] | Promise<Check[]>
}

Pluggability is the point

Verifiers and units are orthogonal. To add a new kind of verification — visual snapshot diff, performance budget, i18n string coverage, dark-mode contrast — you write one file in src/verify/verifiers/, import it in index.ts, and it immediately runs against every existing unit×fixture. Zero component changes. A unit can opt into a subset via verifiers: ["schema", "invariants"]; omit to get all.

6 The runner & verdict taxonomy

src/verify/core/runner.ts is the single execution path. Everything else just calls it. runFixture(unit, fixture, opts?) does:

MountCreate an off-screen container (or use the visible one passed in by UnitPage), createRoot, flushSync(render).
ActIf the fixture has act(), run it with a small driver (click, type, wait). The type helper uses the native value setter so React's onChange fires.
Read contractreadContract(container) → flat {unit, total, done, ...} map.
Run verifiersEvery applicable verifier runs against {unit, fixture, root, contract}. Exceptions become a fail check with the stack as evidence — never swallowed.
Compute verdictAny fail check → FAIL. Mount/setup threw → BLOCKED. Zero fixtures → SKIP. Otherwise PASS.
Return VerifyResult{unitId, fixtureId, verdict, checks[], domSnapshot, durationMs, timestamp, blockedReason?} — fully serializable JSON.

Verdicts

PASSMounted, drove it, every check held. Not "tests pass" or "code looks right" — the surface did what it should.
FAILMounted, drove it, at least one check is fail. The checks[] array tells you exactly which and why.
BLOCKEDCouldn't reach a state where the change is observable — mount threw, no verifiers registered, no container. Not a verdict on the unit. blockedReason says exactly where it stopped.
SKIPNo runtime surface — the unit declares zero fixtures. Nothing went wrong; there's just nothing to observe.

Check statuses

ok ✅ confirmed   fail ❌ observed and wrong   warn ⚠️ concerning but didn't fail   probe 🔍 off-happy-path case that held

When in doubt, FAIL

A false PASS ships broken code; a false FAIL costs one more human look. So: verifier exceptions become fail checks, ambiguous output is FAIL with raw evidence attached, and warn/probe never upgrade a FAIL to PASS.

7 Isolated routes & the dashboard

/verify/:unitId/:fixtureId — the isolated mount

Mounts exactly one unit in exactly one fixture, visibly, on a checkerboard stage. Runs all verifiers against that visible mount and renders a structured report below it. Appending ?chrome=0 hides the harness header/report so an agent can take a clean screenshot of just the component.

This is the "isolated render target" you asked for. No app shell, no providers, no noise. The page also writes the JSON result into <script id="verify-result-json" type="application/json"> so an agent that can only read DOM (no JS eval) still gets structured output.

/verify — the dashboard

Lists every registered unit, its fixtures (with 🔍 markers), its invariants, and which verifiers apply. "Run all" executes the full matrix off-screen and renders a verdict grid. The summary bar (✅ n ❌ n ⛔ n / total) is also emitted as data-verify-summary JSON. There's a "Raw manifest" expander showing exactly what an agent sees from __verify.manifest().

8 window.__verify — the agent handle

Installed at boot by installVerifyHandle(). This is the programmatic surface for the verifier itself — an AI agent, Playwright, or you in devtools.

// What an AI agent does — no DOM scraping, no selector guessing
const manifest = window.__verify.manifest()
// pick the feature slice
const app = manifest.find(m => m.unitId === "todos.feature")
location.href = app.route("add-then-toggle")   // → /verify/todos.feature/add-then-toggle
// ... after navigation ...
const r = window.__verify.current()
r.verdict        // "PASS"
r.checks         // [{verifier:"invariants", status:"ok", label:"after add+toggle, done=1"}, ...]
r.domSnapshot    // {unit:"TodoApp", total:"1", done:"1", active:"0", filter:"all", visible:"1"}

9 One pipeline, three consumers

The most important structural property: there is exactly one way to produce a VerifyResult — runFixture(). The three consumers differ only in how they obtain and present it.

ConsumerEntryMountsPresents
Human (dashboard)/verify → "Run all"Off-screen, in browserVerdict grid + per-unit tables
Human (focused)/verify/:unit/:fixtureVisibly, on stageLive render + report card + JSON sink
AI agent / Playwrightwindow.__verifyOff-screen (runAll) or visible (navigate + current)Raw VerifyResult JSON
CIbun run verify → matrix.test.tsjsdom, headlessvitest pass/fail per unit×fixture

Because it's one pipeline, a regression shows up identically everywhere. You cannot have "passes in CI but the agent sees FAIL" — they're the same function call.

10 End-to-end walkthrough

Trace what happens for todos.feature / whitespace-submit — a 🔍 probe that types only whitespace, force-submits, and asserts no todo was created.

Spec registerstodos.feature.verify.ts calls registerUnit with the fixture {id:"whitespace-submit", probe:true, props:{}, act: type "   " then submit} and invariant whitespace-blocked: contract.total === "0".
Consumer calls runnerDashboard "Run all" / __verify.runAll() / vitest each call runFixture(unit, fixture).
Mount + actTodoApp renders with empty initial. act types " " into the input (native setter → React onChange fires), then dispatches a submit event on the form.
Read contractDOM now shows data-verify-total="0" because useTodos.add() trims and rejects empty.
Verifiers runschema ✅ props valid · dom-contract ✅ contract present · a11y ✅ input labeled · invariants runs 6 predicates incl. whitespace-blocked → probe 🔍.
VerdictNo fail checks → PASS. Result: {verdict:"PASS", checks:[...11], domSnapshot:{unit:"TodoApp", total:"0", ...}}.

The intentional failure: TodoStats / inconsistent-counts

This probe passes {total:10, done:3, active:4}. The Zod .refine(d => d.total === d.done + d.active) fails → schema verifier emits a fail check → verdict FAIL. The matrix test then asserts this is FAIL — proving the framework catches lies, not just confirms truths. If you "fix" the schema to be lax, CI breaks.

11 Extending: new units & verifiers

Add a verifiable unit

// src/verify/specs/MyThing.verify.ts
import { registerUnit } from "../core/registry"
import { MyThing } from "../../features/.../MyThing"

registerUnit({
  id: "MyThing", kind: "component",
  render: (p) => <MyThing {...p}/>,
  propsSchema: z.object({...}),
  fixtures: [
    { id: "default", props: {...} },
    { id: "edge", probe: true, props: {...} },
  ],
  invariants: [{ id: "...", check: ctx => ... }],
})
// then add: import "./MyThing.verify" in specs/index.ts

Add a verifier kind

// src/verify/verifiers/perf.ts
import { registerVerifier } from "../core/registry"

registerVerifier({
  id: "perf",
  description: "Render under 16ms budget",
  run({ root }) {
    const n = root.querySelectorAll("*").length
    return [n < 500
      ? { verifier:"perf", status:"ok",   label:`DOM size ${n}` }
      : { verifier:"perf", status:"warn", label:`DOM size ${n} > 500` }]
  },
})
// then add: import "./perf" in verifiers/index.ts
// — instantly runs against every existing unit×fixture

Neither requires touching any component. That's the modularity claim made concrete.

12 Design principles, restated

  1. Surface, not internals. Verify what's rendered. The DOM contract is the ABI between component and verifier.
  2. One pipeline. runFixture() is the only way to produce a verdict. Dashboard, agent, CI are views on it.
  3. Structured everything. Manifest, results, checks, snapshots — all JSON. An AI agent never has to scrape or guess.
  4. Probes are mandatory. Happy-path-only specs are rejected by CI. At least one 🔍 per unit.
  5. BLOCKED ≠ FAIL. "Couldn't observe" and "observed and wrong" are different facts; conflating them wastes everyone's time.
  6. When in doubt, FAIL. Asymmetric cost. Exceptions become fail checks with evidence, never swallowed.
  7. Orthogonal extension. New units and new verifier kinds are each one file + one import line. They compose without coordination.
