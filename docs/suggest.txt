Với sản phẩm bạn mô tả, câu hỏi đầu tiên không phải là **Frontend + Backend tách riêng hay Monolithic**, mà là:

> **Mô hình dữ liệu có thay đổi liên tục theo nghiệp vụ pháp lý hay không?**

Từ mô tả của bạn, câu trả lời là **Có**.

Ví dụ hôm nay:

* Hợp đồng lao động cần 15 field
* Hợp đồng đại lý cần 23 field
* Đăng ký nhãn hiệu cần 12 field
* Sau 3 tháng bổ sung thêm 8 field
* Sau 6 tháng thay đổi workflow phê duyệt

=> Đây là dạng hệ thống **metadata-driven platform**, không phải CRUD app thông thường.

# Nhưng KHÔNG làm Microservice

Nhiều người sai ở đây.

Sản phẩm này chưa cần:

```txt
Auth Service
Chat Service
OCR Service
Document Service
Legal Service
Workflow Service
```

=> quá sớm.

Tôi sẽ làm:

```txt
Modular Monolith
```

Backend vẫn là 1 application.

Ví dụ:

```txt
backend

modules
 ├── auth
 ├── users
 ├── organizations
 ├── legal-requests
 ├── legal-templates
 ├── document-generator
 ├── workflows
 ├── vault
 ├── compliance
 ├── ai
 └── audit
```

Deploy 1 container.

1 database.

1 codebase.

# Thứ quan trọng nhất: Dynamic Schema

Đây mới là thứ quyết định thành bại.

Nếu hardcode:

```ts
Contract {
  companyName
  taxCode
  representative
}
```

thì sau 6 tháng sẽ phải sửa code liên tục.

---

# Tôi sẽ thiết kế theo Form Definition

Thay vì:

```txt
Contract Form
```

ta lưu:

```txt
Form Definition
```

Ví dụ:

```json
{
  "id": "employment_contract",
  "fields": [
    {
      "key": "employee_name",
      "type": "text",
      "required": true
    },
    {
      "key": "salary",
      "type": "number"
    }
  ]
}
```

Admin có thể thêm field mới.

Không cần sửa code.

---

# Workflow Definition

Tương tự.

Không hardcode:

```txt
Draft
Review
Approved
```

Mà lưu:

```json
{
  "workflow": [
    "draft",
    "legal_review",
    "customer_review",
    "approved"
  ]
}
```

Sau này thêm:

```txt
CEO Approval
```

không phải deploy code.

# Nếu là tôi

Cho sản phẩm Legal-as-a-Service này:

| Thành phần   | Lựa chọn                   |
| ------------ | -------------------------- |
| Frontend     | NextJS                     |
| Backend      | NestJS                     |
| DB           | PostgreSQL                 |
| Architecture | Modular Monolith           |
| Form         | Metadata Driven            |
| Workflow     | Config Driven              |
| Documents    | Template Driven            |
| Storage      | S3/MinIO                   |
| Queue        | Redis                      |
| Search       | PostgreSQL Full Text trước |
| i18n         | vi/en/ja/zh ngay từ đầu    |
| Auth         | RBAC + Organization-based  |
| Audit        | Có từ MVP                  |

Thứ tôi sẽ ưu tiên số 1 không phải chat AI hay OCR.

Mà là:

**Form Definition + Workflow Definition + Document Template Engine.**

Nếu 3 thứ này thiết kế đúng, bạn có thể mở rộng từ 10 nghiệp vụ pháp lý lên 100+ nghiệp vụ mà gần như không phải thay đổi core system. Nếu thiết kế sai, mỗi lần thêm dịch vụ pháp lý mới sẽ phải sửa database, API, UI và deploy lại toàn hệ thống.
