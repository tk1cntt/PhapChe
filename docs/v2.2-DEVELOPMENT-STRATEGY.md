## Problem Analysis

### Current State (v2.2 start)
- ✅ 23 UI screens đã tồn tại (22 pages + 3 layouts)
- 🟡 30-50% functional — UI đẹp nhưng thiếu data
- ❌ Thiếu services, APIs, seed data, validation
- ❌ Thiếu data flow: DB → Service → API → Component

### Root Cause
Mock UI được tạo để demo giao diện, không implement:
1. **Service layer** (business logic trong services, không trong components)
2. **API endpoints** (RESTful theo API_STANDARDS.md)
3. **Seed data** (tất cả data phải từ DB, không hardcode)
4. **State management** (React Query/TanStack Query integration)
5. **Validation** (Zod schemas cho input/output)
6. **Permissions** (RBAC enforcement)
7. **Audit trails** (immutable audit events)

---

## Proposed Approach: Data-First Development

### Philosophy

**Không implement UI-to-UI. Phải có data flow đầy đủ:**

```
Database (Prisma)
    ↓
Service Layer (Business Logic)
    ↓
API Endpoints (RESTful)
    ↓
Frontend Components (React Query)
    ↓
UI Screens (Ant Design)
```

**Mỗi phase phải hoàn thành 4 layers:**
1. **API/Service Layer** — tạo endpoints + business logic
2. **Data Seeding** — tạo seed data cho screen đó
3. **Frontend Integration** — kết nối UI với real API
4. **Tests** — whitebox, blackbox, abnormal, error, E2E

## Development Tiers

### Tier 1: Shared Foundation (Implement trước tất cả phases)

**Lý do:** Nhiều screens share cùng APIs/services. Phải có foundation trước khi làm screen-by-screen.

#### 1.1 Shared APIs & Services

| Domain | Shared APIs | Used By Phases |
|--------|-------------|----------------|
| **Users** | `GET/POST/PATCH /api/admin/users` | 82, 83 |
| **Organizations** | `GET/POST /api/admin/organizations` | 88, 89 |
| **Workspaces** | `GET/POST/PATCH/DELETE /api/workspaces` | 78, 93 |
| **Requests** | `GET/PATCH /api/requests` | 74, 76, 84, 85 |
| **Vault** | `GET/POST /api/vault` | 74, 92 |
| **Audit** | `GET /api/audit` | 74, 79, 81, 90, 91 |
| **Messages** | `GET/POST /api/messages` | 77 |
| **Settings** | `PUT /api/settings/*` | 79, 95 |
| **Partner** | `GET /api/partner/*` | 94, 95 |

**Deliverables:**
- `src/lib/[domain]/services/` — business logic
- `src/app/api/[domain]/` — REST endpoints
- Zod validation schemas
- Permission checks (RBAC)
- Audit logging
- Error handling

#### 1.2 Seed Data Framework

**Lý do:** Tất cả data phải từ DB, không hardcode. Cần seed framework trước khi làm screen-by-screen.

**Deliverables:**
- `prisma/seed/` folder structure
- Per-domain seed scripts (users, orgs, workspaces, requests, etc.)
- `npm run seed` command
- Sample data cho tất cả screens (realistic Vietnamese names, addresses, etc.)
- Relationships preserved (org → workspaces → members → requests)

**Sample Data Requirements:**
- 10 users (customers, specialists, reviewers, admins, partners)
- 3 organizations (with different business types)
- 5 workspaces (across orgs)
- 50 legal requests (various statuses, types, priorities)
- 100 audit events (various actions, actors, targets)
- 20 vault files (with folders, tags)
- 30 messages (across multiple threads)
- 5 partners (with engagements, members)

#### 1.3 Frontend Infrastructure

**Deliverables:**
- React Query providers setup
- Central API client (`src/lib/api/client.ts`)
- Custom hooks (`useRequests`, `useUsers`, `useWorkspaces`, etc.)
- Error boundaries
- Loading states
- i18n translations cho tất cả screens

### Tier 2: Screen-by-Screen Integration (23 phases, 73-95)

**Sau khi Tier 1 hoàn thành, mỗi phase làm:**

1. **Verify APIs exist** (hoặc tạo mới nếu thiếu)
2. **Verify seed data exists** (hoặc thêm nếu thiếu)
3. **Connect UI to real APIs** (thay thế mock data)
4. **Add state management** (React Query hooks)
5. **Add error handling** (API errors, validation errors)
6. **Add loading states** (skeletons, spinners)
7. **Add tests** (whitebox, blackbox, abnormal, error, E2E)

**Example — Phase 76 (My Cases):**
```
Step 1: Verify GET /api/requests exists (from Tier 1)
Step 2: Verify seed data has 50 requests (from Tier 1)
Step 3: Replace mock data in cases/page.tsx with React Query
Step 4: Add search/filter/pagination logic
Step 5: Add error handling (empty state, API errors)
Step 6: Add loading skeletons
Step 7: Add tests (unit + integration + E2E)
```

---

### Tier 3: Cross-Phase Integration (sau khi tất cả phases done)

**Lý do:** Một số flows span multiple screens (create request → my cases → messages).

**Deliverables:**
- End-to-end flow tests (create request, assign specialist, submit for review, etc.)
- State synchronization across screens (optimistic updates)
- Navigation flows (dashboard → cases → details)
- Permission flows (user sees only own data, admin sees all)
- Audit trail completeness (all actions logged)

## Testing Strategy

### Per-Phase Testing (each of 23 phases)

| Test Type | Coverage | Tools |
|-----------|----------|-------|
| **Whitebox** | Services, repositories | Vitest (unit tests) |
| **Blackbox** | API endpoints | Vitest (integration tests) |
| **Abnormal** | Edge cases, empty states | Vitest (boundary tests) |
| **Error** | Error boundaries, fallbacks | Vitest (error tests) |
| **E2E** | Full user flows | Playwright (UI → API → DB) |

**Target:** ≥90% coverage per phase

### Cross-Phase Testing (Tier 3)

- **Create Request Flow:** sign-in → create → my cases → messages
- **Admin Approval Flow:** admin dashboard → request detail → assign → approve
- **Partner Flow:** partner dashboard → request detail → settings
- **Permission Flow:** user sees only own data, admin sees all

## Risk Mitigation

### Risk 1: Mock UI quá đẹp, khó match backend
**Mitigation:** Specification-first approach — phân tích gap trước khi implement. Không cố match 100% UI nếu backend không support.

### Risk 2: Thiếu seed data realistic
**Mitigation:** Seed data framework với realistic Vietnamese data. Mỗi screen có đủ data để test.

### Risk 3: Phases phụ thuộc nhau (shared APIs)
**Mitigation:** Tier 1 (Shared Foundation) implement trước. Mỗi phase chỉ verify APIs exist, không tạo lại.

### Risk 4: Testing coverage khó đạt 90%
**Mitigation:** Mỗi phase phải có tests trước khi mark done. Không skip tests để save time.
