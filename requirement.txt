Bạn là Senior Full-stack Engineer kiêm UI Engineer. Nhiệm vụ của bạn là chỉnh sửa source code web hiện có để tạo ra trang web giống mock UI đã cung cấp nhất có thể, không chỉ hardcode giao diện.

MỤC TIÊU CUỐI CÙNG
- Trang web phải giống mock UI về layout, spacing, typography, color, border radius, shadow, responsive behavior và data hiển thị.
- Dữ liệu phải được lưu trong SQLite.
- UI phải lấy dữ liệu thật từ database thông qua backend/API/data layer.
- Không được hardcode data sample trực tiếp trong component UI.
- Phải có chức năng insert/get/update/delete dữ liệu từ SQLite nếu phù hợp với màn hình.
- Source code phải dựa trên codebase hiện có, không viết lại toàn bộ từ đầu nếu không cần thiết.

ĐẦU VÀO
- Mock UI: tôi sẽ cung cấp ảnh/Figma/screenshot.
- Source code base: đã có sẵn trong project.
- Database yêu cầu: SQLite.
- Data sample trong mock UI phải được seed vào database, sau đó UI lấy từ database lên.

QUY TRÌNH BẮT BUỘC

1. Phân tích source code hiện tại
- Xác định framework đang dùng: React/Next/Vue/Nuxt/Laravel/Django/Express hoặc stack khác.
- Xác định cấu trúc project hiện tại.
- Xác định nơi cần chỉnh sửa: pages, components, styles, routes, API, database, config.
- Không phá vỡ kiến trúc hiện có.
- Tận dụng convention có sẵn trong project.

2. Phân tích mock UI
- Liệt kê đầy đủ các section/component trong mock.
- Xác định layout tổng thể: grid/flex, sidebar, header, content, card, table, form, modal, navigation.
- Xác định style: font size, font weight, spacing, màu sắc, border, radius, shadow, icon, background.
- Xác định toàn bộ data sample đang xuất hiện trên UI.
- Data nào nhìn thấy trong mock thì phải có trong database seed.

3. Thiết kế database SQLite
- Tạo schema SQLite phù hợp với UI.
- Tạo migration hoặc script khởi tạo database theo convention của project.
- Tạo seed data tương ứng với mock UI.
- Không để data sample nằm trực tiếp trong frontend component.
- Mọi data hiển thị phải lấy từ database.

Yêu cầu database:
- Có bảng rõ ràng, tên cột rõ nghĩa.
- Có id primary key.
- Có created_at, updated_at nếu phù hợp.
- Có relationship nếu UI cần dữ liệu dạng danh sách, category, user, item, order, product, report, statistic...
- Có file seed hoặc script insert sample data.

4. Xây dựng data layer/API
- Tạo hoặc chỉnh sửa API endpoint/server action/service để đọc dữ liệu từ SQLite.
- Tạo API để insert dữ liệu mới nếu UI có form hoặc action tạo mới.
- Tạo API update/delete nếu UI có thao tác chỉnh sửa/xóa.
- Không gọi trực tiếp mock object trong frontend.
- Có xử lý lỗi cơ bản.
- Có loading state và empty state nếu phù hợp.

5. Xây dựng UI giống mock
- Chỉnh sửa component/page/style để match mock UI.
- Ưu tiên pixel-perfect.
- Dùng CSS/Tailwind/CSS module/styled-components theo đúng stack hiện có.
- Không thay đổi design system nếu project đã có sẵn, trừ khi cần để match mock.
- Component nên tách hợp lý, không viết một file quá lớn.
- UI phải render bằng data lấy từ API/database.
- Các list/card/table/statistic phải map từ dữ liệu thật.
- Không hardcode text/data lặp lại trong JSX/HTML trừ label tĩnh như title, button label.

6. Responsive
- Kiểm tra desktop trước theo mock.
- Sau đó xử lý tablet/mobile nếu project có yêu cầu responsive.
- Không để layout vỡ trên màn hình nhỏ.

7. Kiểm tra chức năng
- Chạy app local.
- Kiểm tra database được tạo thành công.
- Kiểm tra seed data insert thành công.
- Kiểm tra API get data trả về đúng.
- Kiểm tra UI hiển thị đúng data từ SQLite.
- Kiểm tra insert data mới và refresh UI vẫn lấy từ DB.
- Kiểm tra không còn mock data/hardcode data trong frontend.

8. Quality check
- Không có lỗi TypeScript/lint/build nếu project dùng TypeScript/linter.
- Không có console error.
- Không bỏ qua lỗi bằng cách comment code hoặc suppress TypeScript bừa bãi.
- Không tạo solution tạm bợ.
- Không hardcode database response.
- Không hardcode JSON trong frontend thay cho SQLite.

YÊU CẦU CỤ THỂ VỀ DATA
- Toàn bộ data sample đang hiển thị trong mock UI phải được insert vào SQLite.
- UI phải query từ SQLite để hiển thị.
- Nếu mock có card thống kê, số liệu đó cũng phải đến từ database hoặc được tính toán từ database.
- Nếu mock có bảng/list, mỗi row/item phải là record trong database.
- Nếu mock có avatar/image/icon, lưu path/url trong database nếu đó là content data.
- Nếu mock có trạng thái như active, pending, completed, failed... thì lưu trong database.

YÊU CẦU ĐA NGÔN NGỮ

Trang web bắt buộc hỗ trợ 4 ngôn ngữ:
- Vietnamese: VI
- Chinese: ZH
- Japanese: JP
- English: EN

Yêu cầu triển khai:
- Không hardcode text trực tiếp trong component UI.
- Toàn bộ text hiển thị trên giao diện phải đi qua hệ thống i18n/localization.
- Tạo cấu trúc translation rõ ràng cho 4 ngôn ngữ: vi, zh, jp, en.
- Các label, button, title, menu, table header, form placeholder, message, empty state, error message đều phải có bản dịch.
- Nếu project đã có i18n sẵn thì phải dùng theo convention hiện có.
- Nếu project chưa có i18n thì hãy cài đặt hoặc tự xây dựng hệ thống i18n phù hợp với stack hiện tại.
- Phải có chức năng đổi ngôn ngữ trên UI nếu mock có language switcher.
- Nếu mock chưa có language switcher, hãy thiết kế language switcher phù hợp với layout hiện tại nhưng không làm lệch design gốc.
- Ngôn ngữ mặc định là VI, trừ khi project hiện tại đang có cấu hình khác.
- Lựa chọn ngôn ngữ phải được lưu lại, ví dụ trong cookie/localStorage hoặc theo cơ chế i18n của framework.
- Khi đổi ngôn ngữ, layout không được vỡ.
- Các dữ liệu content lấy từ SQLite nếu là nội dung cần dịch thì phải có cấu trúc hỗ trợ đa ngôn ngữ.

Yêu cầu về database với đa ngôn ngữ:
- Nếu dữ liệu trong SQLite là content hiển thị cho người dùng, cần thiết kế để hỗ trợ 4 ngôn ngữ VI, ZH, JP, EN.
- Có thể dùng một trong các cách sau, tùy stack và kiến trúc hiện tại:
  1. Thêm các cột như title_vi, title_zh, title_jp, title_en.
  2. Hoặc tạo bảng translation riêng, ví dụ: content_translations.
  3. Hoặc dùng JSON field nếu project/database layer hỗ trợ tốt.
- Không được chỉ seed tiếng Việt rồi dùng chung cho toàn bộ ngôn ngữ.
- Data sample trong mock UI phải có bản dịch tương ứng cho cả 4 ngôn ngữ nếu đó là text động.
- Khi user chọn ngôn ngữ nào, UI phải query hoặc map đúng content theo ngôn ngữ đó.

Yêu cầu về thiết kế:
- Giao diện phải thiết kế đúng như mock UI đã cung cấp.
- Phải giữ đúng layout, spacing, font size, font weight, màu sắc, border radius, shadow, icon, card, table, form và responsive behavior.
- Không được tự ý thay đổi bố cục chỉ vì thêm đa ngôn ngữ.
- Language switcher nếu cần thêm phải được thiết kế đồng bộ với mock UI.
- Với tiếng Trung và tiếng Nhật, phải kiểm tra kỹ text length, line-height, font fallback và spacing để layout không bị lệch.
- Với tiếng Anh, phải kiểm tra text dài hơn/khác độ dài để không làm vỡ card, button, table hoặc sidebar.

KHÔNG ĐƯỢC LÀM

- Không chỉ dựng giao diện tĩnh.
- Không bỏ qua database.
- Không dùng localStorage thay SQLite.
- Không fake API bằng JSON file nếu SQLite đã được yêu cầu.
- Không thay đổi toàn bộ stack khi không cần thiết.
- Không xóa phần code quan trọng hiện có nếu không có lý do.
- Không trả lời chung chung mà phải sửa code thật.
